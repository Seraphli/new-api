# Invite Code Registration System

## Overview

This feature adds mandatory invite code registration to new-api. Users must enter a valid, admin-generated invite code to register. This is NOT the same as the built-in "invite link" system — invite codes are generated by the admin and distributed manually, preventing open registration.

## Background

The official QuantumNous/new-api does not support invite code registration (PRs [#1703](https://github.com/QuantumNous/new-api/pull/1703) and [#2936](https://github.com/QuantumNous/new-api/pull/2936) were rejected). This fork adds the feature.

The implementation references [Baoming-qi/New-api-Bom](https://github.com/Baoming-qi/New-api-Bom) (commit `0006f9f`), which implemented invite codes for the classic UI. Since this fork uses the new UI (`web/default/src/`), the frontend was rewritten from scratch. The backend Go code was adapted with minor adjustments.

## How It Works

1. **Admin generates invite codes** via the admin panel (Admin → Invite Codes) or API
2. **Admin distributes codes** to users manually (e.g., via email, chat)
3. **User enters invite code** during registration — the code is consumed atomically with user creation
4. Each invite code can only be used once

## API Endpoints

All endpoints require **Super Admin** (`RootAuth`) authentication.

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/invite-code/` | List all invite codes (paginated) |
| POST | `/api/invite-code/generate` | Generate invite codes (`{"count": N}`, max 100) |
| DELETE | `/api/invite-code/:id` | Delete an invite code |

## Design Decisions

### Why Transaction?

User creation and invite code consumption must be atomic. If the user is created but the invite code fails to be consumed (e.g., race condition with another user), the system would be in an inconsistent state. The transaction with `FOR UPDATE` row lock ensures exactly one user can consume a given code.

### Why Hardcoded `invite_code_required: true`?

The current implementation always requires invite codes (`invite_code_required: true` is hardcoded in `controller/misc.go`). To make this configurable via admin settings, add a system option and read it in `GetStatus`. This was kept simple intentionally — if open registration is needed, set it to `false` or remove the line.

### Frontend Conditional Display

The invite code input field only appears when `status.invite_code_required` is `true`. This means if the hardcoded value is changed to a configurable setting in the future, the frontend will work without changes.

## Upstream Sync Notes

When syncing with upstream (`git fetch upstream && git merge upstream/main`):

- **New files** will not conflict (they don't exist upstream)
- **Low-risk files** are append-only additions to lists/configs — easy to resolve
- **`controller/user.go`** is the main conflict risk — if upstream modifies the `Register` function, manual merge is needed. The key change: `Insert()` → `Transaction(InsertWithTx + ConsumeInviteCodeTx)`
- **Sidebar config files** (`model/user.go`, `controller/user.go`) may conflict if upstream adds new sidebar modules — resolution is straightforward (add `invite_code` alongside whatever upstream added)
